# Информационная система цифрового автосалона (Car Dealership Microservices)

Проект представляет собой отказоустойчивую распределенную backend-систему, разработанную с нуля и прошедшую эволюцию от монолитного ядра с in-memory хранилищем до современной микросервисной архитектуры.

Система автоматизирует полный цикл работы автосалона: управление складскими остатками и запчастями, конфигурацию автомобилей под индивидуальные требования клиентов, обработку заказов, учет тест-драйвов и разграничение прав доступа для различных ролей пользователей.

---

## Архитектура и стек технологий

Проект построен на базе микросервисной архитектуры с четким разделением контекстов и обязанностей, распределенным управлением данными и надежным межсервисным взаимодействием.

### Основной технологический стек:
* **Язык программирования:** Java
* **Фреймворки:** Spring Boot, Spring Data JPA, Spring Security
* **Базы данных и кэширование:** PostgreSQL, Redis
* **Межсервисное взаимодействие:** gRPC (синхронное), Брокер сообщений (асинхронное)
* **Миграции данных:** Liquibase
* **Безопасность и IAM:** Keycloak (OAuth2 / OpenID Connect)
* **Сборка и контейнеризация:** Gradle, Docker, Docker Compose
* **Тестирование:** JUnit 5, Testcontainers

## Ключевые модули и сервисы системы

### 1. StorageService (Сервис склада)
Отвечает за учет автомобилей в наличии, управление номенклатурой запчастей и отслеживание складских остатков. 
* Включает встроенный высокопроизводительный **gRPC-сервер**, который поставляет актуальные данные о доступных к продаже машинах другим сервисам.
* Использует изолированную базу данных PostgreSQL.

### 2. OrderService (Сервис заказов и конфигуратора)
Управляет жизненным циклом заказов и предоставляет клиентский REST API. 
* Реализует **умный конфигуратор автомобилей**, позволяющий собрать кастомную комплектацию (колеса, трансмиссия, интерьер и т.д.) на основе базовой модели, динамически рассчитывая стоимость и строго проверяя совместимость узлов по бизнес-правилам.
* Выступает в роли **gRPC-клиента**, запрашивая у `StorageService` оперативную информацию о наличии машин на складе.

## Продемонстрированные инженерные навыки

Этот проект наглядно иллюстрирует владение ключевыми концепциями современной backend-разработки:

### Распределенные системы и интеграция
* **Синхронное взаимодействие (gRPC):** Проектирование контрактов в `.proto` файлах, генерация кода и реализация связки gRPC Client-Server между сервисами.
* **Отказоустойчивость (Fault Tolerance):** Настройка таймаутов на стороне клиента, логирование межсервисного взаимодействия и безопасная обработка недоступности систем с возвратом контролируемых ошибок (503 Service Unavailable).

### Работа с базами данных
* **Проектирование хранения:** Работа с реляционными базами данных, автоматическое ведение схемы через миграции Liquibase и наполнение базы начальными (seed) данными.
* **Оптимизация запросов:** Реализация гибких и динамических фильтров для каталога автомобилей (по цене, бренду, типу кузова, двигателю и т.д.) на уровне базы данных с помощью Spring Data Specifications.
* **Безопасное удаление:** Использование концепции мягкого удаления (Soft Delete) для сохранения целостности исторических данных.

### Безопасность (Security)
* **Централизованная авторизация:** Полная интеграция со Spring Security и Keycloak. Собственная база пользователей отсутствует - аутентификация делегирована identity provider'у.
* **Ролевая модель (RBAC):** Разграничение доступа к REST API для четырех ролей: Клиент (USER), Менеджер (MANAGER), Администратор склада (WAREHOUSE_ADMIN) и Системный администратор (ADMIN).
* **Контроль на уровне методов:** Использование аннотаций `@PreAuthorize` для проверки владения ресурсом. Например, обычный пользователь имеет доступ на просмотр и отмену только своих заказов, тогда как менеджеры и админы могут управлять всеми сущностями.

### Качество кода и архитектурный дизайн
* **Чистый код:** Соблюдение принципов SOLID, декомпозиция логики на три слоя (контроллеры, сервисы, репозитории) и строгое разделение внутренних JPA-сущностей и внешних данных с помощью DTO.
* **Покрытие тестами:** Надежность системы подтверждена комплексными интеграционными тестами с использованием Testcontainers, которые поднимают реальный Docker-контейнер PostgreSQL прямо во время тестирования для проверки миграций и API-слоя.

## Локальный запуск

Для запуска всей инфраструктуры и сервисов вам понадобятся установленные Docker и Docker Compose.

1. Соберите проект с помощью Gradle:
   ./gradlew clean build -x test

2. Запустите окружение (PostgreSQL, Keycloak и микросервисы):
   docker-compose up --build

3. После успешного запуска интерактивная документация API будет доступна по адресу:
   http://localhost:8080/swagger-ui.html

## Дополнительно: Инструмент управления файловой системой (CLI Utility)
В рамках развития архитектурных навыков (в частности, паттернов проектирования и принципов SOLID) в репозитории также представлена изолированная консольная утилита для работы с файловой структурой. 

Она работает независимо от внешних библиотек, поддерживает unix-синтаксис путей, навигацию и базовые файловые операции (копирование, перемещение, переименование). В ней реализовано ленивое (lazy) объектное представление данных для экономии оперативной памяти и абстракция от конкретного типа файловой системы (соблюдение принципа Open-Closed Principle).
