# Car Dealership Microservices

Этот pet-проект представляет собой backend-систему цифрового автосалона, эволюционировавшую от монолита к микросервисной архитектуре. Система автоматизирует конфигурирование автомобилей, оформление заказов, тест-драйвы, складской учет и внутреннюю сборку.

## О проекте

Этот проект прошел полный цикл эволюции от монолитного приложения с in-memory хранилищем до современной отказоустойчивой микросервисной архитектуры с распределенным управлением данными, асинхронным событийно-ориентированным и синхронным межсервисным взаимодействием.

Система автоматизирует бизнес-процессы автосалона: онлайн-конфигуратор автомобилей под требования клиента, оформление заказов на машины в наличии и под заказ, учет заявок на тест-драйв, управление складскими остатками и внутреннюю сборку автомобилей.

## Архитектура

### Микросервисы



### 1. OrderService (Сервис заказов)
Центральная точка взаимодействия с клиентами и менеджерами.
* **Зона ответственности:** Оформление заказов на покупку готовых машин, работа со спецификациями кастомных комплектаций, регистрация заявок на тест-драйв, автоматическое назначение менеджеров и контроль жизненного цикла клиентского заказа.
* **Конфигуратор:** Содержит логику базовых комплектаций моделей и альтернативных узлов (колеса, трансмиссия, руль, интерьер). Рассчитывает итоговую стоимость и валидирует совместимость компонентов по строгим бизнес-правилам доменной области.
* **Интеграция:** Публикует события в брокер при оплате заказов, а также выступает в роли gRPC-клиента для быстрого запроса актуального каталога доступных машин.

### 2. StorageService (Сервис склада)
Отвечает за учет материальных ценностей автосалона и внутреннюю логистику.
* **Зона ответственности:** Ведение номенклатуры автомобилей и автозапчастей, отслеживание складских остатков и резервов. Бизнес-логика складского учета полностью отделена от конфигуратора через стабильные идентификаторы деталей.
* **Заказ на сборку (Assembly Order):** Содержит внутреннюю сущность для отслеживания складского процесса подготовки машины под конкретный клиентский заказ (подбор запчастей, резервирование, сборка комплектации).
* **Интеграция:** Слушает события брокера от `OrderService`, запускает процессы сборки, отправляет обратно результаты выполнения операций, а также выступает в качестве gRPC-сервера.

## Отказоустойчивость и межсервисное взаимодействие

### Transactional Outbox

Для гарантированной доставки сообщений используется паттерн **Transactional Outbox**. Изменение состояния заказа и запись события в таблицу outbox происходят в рамках одной транзакции, после чего отдельный процесс публикует событие в брокер сообщений. Это исключает потерю сообщений при сбоях приложения или сети.

### Apache Kafka

Длительные бизнес-процессы реализованы через событийную архитектуру. После оплаты заказа `OrderService` публикует событие в Kafka/RabbitMQ, которое обрабатывается `StorageService`. Ответные события переводят заказ в следующее состояние или отменяют его. Обработка сообщений реализована идемпотентно.

### gRPC

Для операций, требующих минимальной задержки, используется синхронное взаимодействие через gRPC. `OrderService` получает актуальную информацию о наличии автомобилей непосредственно из `StorageService`. На стороне клиента настроены таймауты и обработка ошибок, поэтому при недоступности склада API корректно возвращает ошибку `503 Service Unavailable` вместо зависания запроса.


## Инженерные решения



### Разделение на микросервисы и распределенные транзакции
* **Асинхронное взаимодействие:** Реализован событийно-ориентированный процесс. При переходе заказа в статус "Оплачен", `OrderService` отправляет сообщение брокеру. `StorageService` обрабатывает его, резервирует ресурсы, формирует заказ на сборку и отправляет подтверждение или отказ, переводя клиентский заказ в финальное состояние.
* **Паттерн Outbox:** Для обеспечения гарантированной доставки сообщений и атомарности (чтобы изменение статуса в БД и отправка события в брокер происходили в одной транзакции) реализована таблица outbox. Это исключает потерю сообщений при сбоях сети или брокера.
* **Идемпотентность:** Обработка входящих сообщений защищена от дублирования со стороны брокера.
* **Синхронный gRPC:** Для мгновенного отображения каталога машин в наличии, REST API сервиса `OrderService` запрашивает актуальные данные у gRPC-сервера `StorageService` по сгенерированным из `.proto` файлов контрактам.
* **Отказоустойчивость (Fault Tolerance):** На gRPC-клиенте настроены таймауты. В случае полной недоступности склада, система корректно обрабатывает ошибку и возвращает пользователю статус `503 Service Unavailable`.

### Проектирование и оптимизация работы с БД
* **Изоляция данных:** Каждый сервис работает исключительно со своей базой данных PostgreSQL, исключая сквозные запросы.
* **Управление миграциями:** Инфраструктура таблиц и наполнение начальными (seed) данными автоматизированы через миграционные скрипты Liquibase.
* **Динамические фильтры:** Каталог и поиск комплектаций оптимизированы с помощью Spring Data Specifications, что позволяет выполнять гибкую фильтрацию параметров (цена, бренд, мощность, тип КПП) непосредственно на уровне запросов к СУБД.
* **Мягкое удаление (Soft Delete):** Базовые сущности поддерживают флаг удаления, что сохраняет исторические данные для аналитики без нарушения ссылочной целостности.

### Интеграция безопасности (Identity & Access Management)
* **Делегирование аутентификации:** Проект интегрирован с сервером Keycloak, развернутым в Docker. Вся проверка подлинности пользователей вынесена за пределы бизнес-логики.
* **Ролевая модель доступа (RBAC):** REST API разделен на уровни доступа для четырех ролей:
  * `USER` (Клиент) - создание заказов, управление только своими заявками.
  * `MANAGER` (Менеджер) - сквозной просмотр всех заказов и ведение клиентов.
  * `WAREHOUSE_ADMIN` (Администратор склада) - управление заказами на сборку и остатками.
  * `ADMIN` (Администратор системы) - полный доступ ко всей инфраструктуре.
* **Декларативная защита методов:** Использование аннотаций `@PreAuthorize` в связке со Security Context для автоматической проверки того, что обычный пользователь не может просмотреть или отменить чужой заказ.

### Качество кода и тестирование
* **Чистая архитектура:** Четкое разделение слоев (контроллеры, бизнес-сервисы, JPA-репозитории). Для безопасной передачи данных за пределы сервисов используются DTO, маппинг которых автоматизирован.
* **Сквозное логирование:** В распределенные процессы внедрен корреляционный идентификатор (`traceId`), позволяющий отслеживать путь бизнес-транзакции по логам разных микросервисов.
* **Тестирование в реальном окружении:** Покрытие кода интеграционными тестами с использованием Testcontainers. Базы данных PostgreSQL, инфраструктура gRPC и брокеры сообщений автоматически поднимаются в изолированных Docker-контейнерах во время прогона тестов для полной симуляции production-среды.



## Локальный запуск



Для развертывания системы вам понадобятся установленные Docker и Docker Compose.

1. Соберите исполняемые файлы микросервисов с помощью Gradle:
   ./gradlew clean build -x test

2. Запустите всю инфраструктуру (базы данных, брокер, Keycloak и сами сервисы) одной командой:
   docker-compose up --build

3. Интерактивная документация для проверки эндпоинтов и тестирования API (Swagger UI) будет доступна по адресу:
   http://localhost:8080/swagger-ui.html
